Trigent Plugins — Installation Guide & Playbook

Install and use the Trigent Claude Code / Cursor plugins from the trigent-plugins marketplace. This guide covers the shared install flow, then a dedicated section for each plugin.

Marketplace: trigent-plugins Claude Code · Cowork · Cursor aidlc v4.4.0 trigent-pid v1.1.0

Overview

The trigent-plugins marketplace bundles two governed-delivery plugins. This guide documents both.

aidlc — AI-Driven Development Lifecycle

Nine chained commands that take work from a goal to tracked Jira/Linear artifacts and TDD implementation, using Jira terminology (Feature → Epic → Story → Task, scheduled in Sprints).

trigent-pid — Project Initiation Document

Guides creation and iterative refinement of a Project Initiation Document (business case + solution design) in Confluence, through to final approval.

One marketplace, install what you needBoth plugins live in the same repo and are installed from the same marketplace. You can install one or both.

Add the marketplace & install

1. Add the marketplace

claude plugin marketplace add git@gitlab.com:trigent/aidevops/claude-plugins.git

Or from inside a Claude Code session:

/plugin marketplace add git@gitlab.com:trigent/aidevops/claude-plugins.git

From a local checkout instead (point at the folder that contains .claude-plugin/marketplace.json):

/plugin marketplace add D:\Project-Dev-Space\CodexAI\iGenFactory\Tri-AIDLC-Plugin\claude-plugins-main

2. Install the plugin(s)

# the AI-DLC lifecycle plugin
/plugin install aidlc@trigent-plugins

# the PID (Project Initiation Document) plugin
/plugin install trigent-pid@trigent-plugins

Reload when prompted, then confirm the commands are available by typing /.

Using in Cursor

Cursor reads rule files (.mdc) from a workspace's .cursor/rules/ folder rather than Claude SKILL.md files. The aidlc plugin ships a ready-made Cursor layer under plugins/aidlc/.cursor/. Copy it into your project:

mkdir -p .cursor/rules
cp -r plugins/aidlc/.cursor/rules/* .cursor/rules/
cp plugins/aidlc/aidlc.config.example.yaml ./aidlc.config.example.yaml

Reload Cursor — the always-on lifecycle gate applies automatically; invoke a command by referencing it (e.g. "run aidlc-intent").

aidlc Plugin

AI-DLC is a governed lifecycle: capture the goal, decompose it, design it, verify readiness, build test-first, and review. AI proposes at each step; a human approves before the workflow advances. This edition uses standard Jira terminology and reads all organisation settings from aidlc.config.yaml — nothing is hardcoded.

Anytime commands/aidlc-help, /aidlc-progress, and /aidlc-init can run at any point; the rest follow the phase order.

Concepts & hierarchy

Feature the goal / initiative — created by /aidlc-intent aidlc:feature └── Epic self-contained, loosely coupled work area aidlc:epic └── Story user story ("As a… I want… so that…") aidlc:story └── Task implementation task / sub-task (Task Spec) aidlc:task aidlc:sprint Sprint = groups Stories/Tasks for scheduling (not a hierarchy level)

Configuration — /aidlc-init

Run /aidlc-init once per project. It asks a few questions and writes aidlc.config.yaml to your project root; every other skill reads it, so no organisation details are hardcoded.

backend:
  default: "gitlab"          # gitlab | linear | confluence
atlassian:
  cloudId: ""                # e.g. yourcompany.atlassian.net
confluence:
  spaceKey: ""               # e.g. ENG
  featureIndexPageId: ""     # blank on first run; auto-created
jira:
  projectKey: ""             # e.g. PROJ
gitlab:
  docsRepoUrl: ""            # https://gitlab.com/org/group/ai-dlc-docs
  docsRepoSsh: ""
  docsPath: ""               # or $AIDLC_DOCS_PATH
linear:
  teamKey: ""
links:
  performanceStandards: ""

These resolve the placeholder tokens used across the skills: <ATLASSIAN_CLOUD_ID>, <CONFLUENCE_SPACE_KEY>, <FEATURE_INDEX_PAGE_ID>, <GITLAB_DOCS_REPO_URL>, <GITLAB_DOCS_REPO_SSH>, <PERFORMANCE_STANDARDS_URL>.

Keep secrets out of gitIf aidlc.config.yaml holds internal URLs/keys, add it to .gitignore; commit the aidlc.config.example.yaml template instead.

trigent-pid Plugin

The trigent-pid plugin guides a team through creating or refining a Project Initiation Document (PID) — a structured Confluence page capturing the business case and solution design for significant projects, features, risky refactors, or large cross-team initiatives. It's a single command, /trigent-pid, that runs an AI-led interview, publishes to Confluence, and manages comment-resolution rounds through to approval.

Install & invoke

/plugin install trigent-pid@trigent-plugins

# then, in a session:
/trigent-pid

On start it asks which mode you want:

ModeWhat happens
New PIDAI interviews you for the Business Case, does market/competitive research, cross-references related initiatives, drafts and iterates, then publishes to Confluence — optionally continuing into the Solution Design.
Refine existing PIDAI reads the published page's comments and runs a comment-resolution round: proposes resolutions, applies accepted changes, records what was addressed/deferred.

Configuration — space-agnostic placeholders

The plugin is not tied to any single Confluence space. Three placeholders must be resolved before any Confluence operation (the cloudId is fetched at runtime automatically):

PlaceholderMeaning
<CONFLUENCE_SPACE_KEY>Target Confluence space key (e.g. ENG)
<CONFLUENCE_SPACE_ID>Numeric space ID for that space
<PID_PARENT_PAGE_ID>Page ID of the parent folder PIDs publish under (e.g. "Business Cases — In Review")

They resolve in this order of preference:

  1. From a pid.config.yaml in the project root (template: pid.config.example.yaml in the plugin).
  2. If an aidlc.config.yaml exists, reuse its atlassian.cloudId and confluence.spaceKey.
  3. Otherwise, the skill asks for the space key, resolves the space ID via getConfluenceSpaces, and asks which parent page PIDs should live under.
# pid.config.yaml
atlassian:
  cloudId: ""              # usually auto-detected at runtime
confluence:
  spaceKey: ""             # -> <CONFLUENCE_SPACE_KEY>  e.g. "ENG"
  spaceId: ""              # -> <CONFLUENCE_SPACE_ID>   numeric
  pidParentPageId: ""      # -> <PID_PARENT_PAGE_ID>    parent folder page id

Playbook

/trigent-pidchoose mode→ Business Caseinterview + research→ Publishto Confluence→ Solution Designoptional→ Refineresolve comments→ Approval
  1. Configure the Confluence space once (pid.config.yaml) or let the skill prompt you.
  2. New PID → answer the Business Case interview; the AI researches the market and cross-references related initiatives, drafts, and iterates to your approval.
  3. Publish the draft to your configured Confluence space/parent folder; share the URL with stakeholders for comments.
  4. Solution Design (optional) continues the interview and drafting for the technical design.
  5. Refine → run /trigent-pid again, choose "Refine existing PID", and the AI resolves reviewer comments round by round until final approval.
Human-in-the-loopLike aidlc, the AI proposes and you approve — nothing is published or resolved without your confirmation.

Troubleshooting

SymptomFix
Commands don't appear after installReload the IDE/session; confirm the marketplace was added and /plugin install <name>@trigent-plugins succeeded.
aidlc: unresolved placeholder (e.g. <CONFLUENCE_SPACE_KEY>)Run /aidlc-init — the value is blank in aidlc.config.yaml.
trigent-pid: "which space?" / unresolved space placeholderCreate pid.config.yaml from the example, or let the skill prompt you for the space key.
Confluence write failsConfirm the Atlassian account has edit permission on the target space and that the space key/ID are correct.
aidlc GitLab: repo not cloned / glab missingSet gitlab.docsPath (or $AIDLC_DOCS_PATH), install glab, run glab auth login.
Cursor doesn't pick up the workflowEnsure the .mdc files are in your project's .cursor/rules/ and reload Cursor.
